# Fathom

<img src="logo.png" width="200" height="200" alt="Fathom" />

A macOS app that lets you semantically search files on your computer — like a search engine, but for your local documents.

Fathom indexes files by splitting them into sentence-level chunks, grouping related chunks semantically, and storing vector embeddings in a FAISS index backed by SQLite metadata. At search time, the query is embedded and matched against the index using vector similarity, returning the most relevant passages along with their source file.

## How It Works

### Indexing

1. User selects a directory on their computer
2. Python scripts (`index.py` and `summarize.py`) iterate through all files, split text into sentence-level chunks, and generate embeddings for each chunk
3. Chunks are grouped by semantic similarity; the most representative sentence from each chunk is identified
4. Each chunk's best sentence is added to a FAISS vector index, with metadata (file name, path, chunk number) stored in SQLite

### Querying

1. User types a search query
2. The query is vectorized and the FAISS index is searched for the most similar chunks
3. Results are returned with file name, path, and chunk index
4. User can click a result to open the file in its default application

### Language Model

The embedding model is hosted two ways:
- **Local development:** `model_server.py` serves it on `localhost:5000`
- **Production:** `model_service.py` runs it internally within the Mac app

## Technical Deep Dive

### Fragmentation and Chunking

Each file is processed by `SemanticFragmenter`, which splits text into sentence-level fragments. Each fragment is tracked by its position in the document (`fragment_index`, `fragment_sub_index`) and length, and hashed for change detection. Fragments are filtered to remove very short strings before embedding.

### Clustering

Fragment embeddings are grouped into semantic clusters using k-means. Each cluster represents a coherent topic or idea within the document. For each cluster, the fragment whose embedding is closest to the cluster centroid is identified as the canonical representative. Only these representative embeddings (`center_embeddings`) are added to the FAISS index — keeping the index compact regardless of document length.

### Dimensionality and Vector Index

Embeddings are generated by a local language model (`EmbeddingModel`) and stored as float32 vectors. FAISS is used for approximate nearest-neighbor search over these vectors. Each vector in the index is keyed to a SQLite row ID in the `fragments` table, which stores the document position metadata needed to locate the passage at query time.

### Indexing Pipeline

Files move through a state machine tracked in SQLite (`FilenamesDB`): `queued → processing → indexed` (or `failed`). Up to 2 files are processed concurrently in daemon threads. A central event-driven loop (`run_indexing_loop`) wakes on a `threading.Event` signal — fired when a file finishes processing or a new file is detected — and dispatches the next queued file if a thread slot is free. Modified files are automatically re-queued; failed files are retried on the next run.

## Dependencies

- [ebooklib](https://github.com/aerkalov/ebooklib) — EPUB parsing
- [BeautifulSoup4](https://www.crummy.com/software/BeautifulSoup/) — HTML cleaning
- [python-docx](https://python-docx.readthedocs.io/) — DOCX parsing
- [striprtf](https://github.com/joshy/striprtf) — RTF parsing
- [nltk](https://www.nltk.org/) — Sentence tokenization
- [FAISS](https://github.com/facebookresearch/faiss) — Vector similarity search
- antiword (system dependency, for `.doc` files)

```bash
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt
```

```bash
python app.py
```
